# Client Architecture

This document describes the AutoTech Client architecture, its purpose, installation model, and interaction with the AutoTech Web Dashboard. This is a technical reference only. Governance and enforcement rules are defined in `CLAUDE.md`.

---

## 1. Overview

AutoTech uses a **client-side helper application** to correctly execute tools such as SSH, SFTP, VNC, and batch scripts on a user’s local machine. The client is required for proper remote operation when accessing the dashboard from another PC.

Without the client installed, all tool launches occur on the **server machine**, which is incorrect for most operational use cases.

---

## 2. Problem the Client Solves

### Without AutoTech Client
- SSH, VNC, and file tools execute on the server
- Tools open in the server’s desktop/session
- Remote users cannot interact with equipment locally

### With AutoTech Client
- Tool execution is redirected to the user’s local PC
- SSH/VNC sessions open locally
- File transfer tools run with local filesystem access

This separation is fundamental to AutoTech’s design.

---

## 3. Custom URI Handler Model

The client registers **custom URI schemes** with Windows. These URIs are generated by the web UI and invoked by the browser.

### Supported URI Schemes

- `autotech-ssh://user@host`
- `autotech-sftp://user@host`
- `autotech-vnc://host`
- `autotech-script://ScriptName.bat`

When a URI is clicked:
1. Browser resolves the custom protocol
2. Windows launches the registered handler
3. AutoTech Client executes the mapped local tool

---

## 4. Client Responsibilities

The AutoTech Client is responsible for:
- Registering URI handlers with Windows
- Launching local SSH clients (e.g. PuTTY)
- Launching local SFTP clients (e.g. WinSCP)
- Launching local VNC viewers
- Executing approved batch scripts
- Reporting successful installation to the server

The client does **not**:
- host a web server
- maintain persistent connections
- expose services
- store credentials long-term
